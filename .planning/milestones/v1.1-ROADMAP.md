# Milestone v1.1: Guided Onboarding Walkthrough

**Status:** ✅ SHIPPED 2026-06-19
**Phases:** 1–6
**Total Plans:** 18

## Overview

v1.1 replaces SweatPact's front-loaded four-screen setup wizard with a minimal
mandatory start (username only) plus a contextual coachmark walkthrough that
teaches the app and completes optional setup in-context. The build was strictly
dependency-ordered: server-side persistence and pure step/completion logic landed
first; shared action surfaces were extracted before any coachmark embeds them;
the `TourProvider` was wired and resume-verified with no coachmarks before the
spotlight engine was built; the single-route coachmark engine was hardened before
cross-route sequencing and teaching content rode on top of it; skip-already-done
and replay hardening landed last. North star: the user is in the real app
immediately and the walkthrough teaches the four points (gym, stakes challenge,
money model, iOS Shortcut) by doing real in-context actions — including one
clearly-labeled practice check-in that never affects real stakes.

## Milestone Summary

**Timeline:** 2026-06-14 → 2026-06-19 (5 days)
**Files changed:** 171 files, +19,298 / -512 lines

**Key Accomplishments:**
1. New user enters the real app via username-only (no wizard) — ONB-01 shipped
2. Server-side onboarding progress persisted in `onboarding_progress` table with owner-only RLS — PROG-01
3. Full 5-step coachmark walkthrough runs end-to-end across all tabs with navigate-then-reveal sequencing — TOUR-05
4. Practice check-in has hard financial safety guarantee: zero `/api/checkin` calls (grep gate) — TEACH-05
5. Already-done steps auto-skipped from real app state (gymCount + restDays) — PROG-02
6. Brand-voiced "pact is live" completion moment shipped; legacy wizard fully deleted — UX-03

**Key Decisions:**
- `onboarding_progress` is the runtime source of truth for tour state; `profiles.onboarding_complete` read only at backfill (D-03)
- PATCH endpoint accepts a single semantic `complete_step` key — dedupe-appended server-side, never a client full array (D-04)
- TOUR_VERSION=1, ordered STEPS [schedule,gym,challenge,money,shortcut_viewed]; schedule is setup-bearing but NOT a completion-gating teaching key
- Completion probes derive from real state (gymCount, rest_days non-empty, completed_steps) — no duplicate flag (PROG-02)
- react-joyride v3.1 chosen for coachmark engine; Radix-dialog anchoring via pause/hide (portal-within-portal deferred beyond Phase 5)
- Practice check-in is cosmetic only: zero fetch, zero /api/checkin; only side effect is handleAdvance() (TEACH-05/D-05)
- Real gymCount + restDays probe flows server-side from (tabs) layout RSC into deriveCurrentStep — no new client fetch (D-07)
- Legacy /onboarding/{gym,schedule,shortcut} wizard pages deleted at close (D-08/D-09)

**Issues Resolved:**
- Per-page onboarding bounces removed from all 8 tab pages — single layout gate (Phase 3)
- Radix-dialog collision with coachmark overlay resolved via pause/hide pattern (Phase 4)
- Invited-path route swap (self-starter vs. invited) resolved via `data-pending-count` DOM bridge (Phase 5)

**Issues Deferred:**
- Browser runtime verification of navigate-then-reveal sequencing (requires real device)
- Invited-path runtime confirmation (requires pending DB invite)
- Practice check-in Network-tab zero-request proof deferred to production (user decision, STATE.md)
- UAT sessions for Phases 03, 04, 06 partially skipped by user at execution time

**Technical Debt Incurred:**
- `as any` casts in CoachmarkRenderer where joyride types are incomplete (acceptable — library boundary)
- Three `eslint-disable react-hooks/exhaustive-deps` suppression comments in coachmark-renderer.tsx (all documented with rationale)
