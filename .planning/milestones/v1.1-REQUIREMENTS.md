## v1.1 Requirements — Guided Onboarding Walkthrough

### Onboarding Entry & Flow

- [x] **ONB-01**: New user completes a minimal mandatory start (username only) and lands directly in the real app — Phase 3 ✓
- [x] **ONB-02**: The `(tabs)` redirect gate no longer forces the full setup wizard — only a missing username redirects; optional setup is deferred into the walkthrough — Phase 3 ✓
- [x] **ONB-03**: Walkthrough supports both entry paths — self-starter (start a challenge) and invited (accept a partner's challenge invite) — Phase 5 ✓
- [x] **ONB-04**: User can skip the walkthrough at any step without being blocked or nagged, and keep using the app — Phase 3 ✓

### Coachmark Walkthrough Engine

- [x] **TOUR-01**: Contextual coachmarks spotlight live UI elements one at a time, and only show once the target element is actually mounted (no spotlighting empty/streamed space) — Phase 4 ✓ (MutationObserver anchor gate)
- [x] **TOUR-02**: The coachmark overlay is click-through (cutout) and coexists with the nav stack, Radix dialogs, and the install gate without trapping input or hiding behind chrome — Phase 4 ✓ (z-[100] in #tour-root, Radix pause)
- [x] **TOUR-03**: Coachmarks position correctly within PWA safe-area insets on mobile standalone — Phase 4 ✓ (safe-area env() CSS)
- [x] **TOUR-04**: Coachmarks are accessible — keyboard advance/skip/dismiss, focus handling, and reduced-motion support — Phase 4 ✓ (aria-live, prefers-reduced-motion)
- [x] **TOUR-05**: The walkthrough sequences across tabs/routes (navigates, then reveals the next step once its anchor is ready) — Phase 5 ✓

### Teaching Steps

- [x] **TEACH-01**: The walkthrough teaches and completes gym setup in-context (Google Places) — Phase 5 ✓
- [x] **TEACH-02**: The walkthrough teaches starting (or accepting) a stakes challenge in-context — Phase 5 ✓
- [x] **TEACH-03**: The walkthrough teaches the money model — earned/owed, penalties, settlement — anchored to real UI — Phase 5 ✓
- [x] **TEACH-04**: The walkthrough teaches the iOS Shortcut integration, with manual check-in shown as the universal fallback for non-iOS users — Phase 5 ✓
- [x] **TEACH-05**: The first walkthrough check-in is a clearly-labeled practice check-in that does NOT register as a real check-in or affect stakes, penalties, or stats — Phase 5 ✓ (grep gate: 0 api/checkin refs in executable code)
- [x] **TEACH-06**: The walkthrough is considered complete once all four teaching points (gym, challenge, money, Shortcut) have been presented/done — Phase 2 ✓

### Setup-as-Action Surfaces

- [x] **SETUP-01**: Gym, schedule, and Shortcut setup UIs are reusable surfaces callable from both the walkthrough and the legacy entry, hitting the existing endpoints (no logic fork) — Phase 2 ✓
- [x] **SETUP-02**: User can set their weekly schedule / goal in-context during the walkthrough — Phase 5 ✓

### Progress & Persistence

- [x] **PROG-01**: Walkthrough progress is persisted server-side per user, so it resumes after interruption and across devices — Phase 1 ✓
- [x] **PROG-02**: Steps already completed are auto-skipped, derived from real app state (gym set, weekly goal set, Shortcut viewed) rather than a duplicate flag — Phase 2 (logic) + Phase 6 (live probe wiring) ✓
- [x] **PROG-03**: User can replay the walkthrough anytime from Settings — Phase 6 ✓
- [x] **PROG-04**: Replay handles walkthrough version changes gracefully without breaking on stale/removed step targets — Phase 1 (tour_version field) + Phase 6 (version-drift replay) ✓

### Onboarding UX

- [x] **UX-01**: A 4-item "getting started" checklist shows progress and completes as the real actions are done — Phase 5 ✓
- [x] **UX-02**: The dashboard shows a "Start your first pact" empty-state CTA as a fallback for users who skip coachmarks — Phase 5 ✓
- [x] **UX-03**: A sharp, brand-voiced "pact is live" completion moment marks walkthrough/first-challenge completion — Phase 6 ✓
- [x] **UX-04**: Walkthrough copy is outcome-framed and brand-voiced (consequence-first, "stakes not stats") — Phase 5 ✓
