Build the server-only PostHog Query API client and the typed HogQL builders/parsers for the three analytics panels: onboarding funnel (DASH-04), feature adoption (DASH-05), engagement & retention (DASH-06). `runHogQL()` is a raw `fetch()` (NOT posthog-node) so results land in Next's Data Cache via `next: { revalidate: 3600 }`, respecting the PostHog Query API rate limit. Purpose: PostHog data is read here once and cached; correct host + fail-soft behavior are what keep the admin page resilient when analytics is unavailable.
Output: `src/lib/admin-posthog.ts` (client + builders + Zod parsers) + tests + `.env.example` updates. No React.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| server → PostHog Query API | Personal API key is a server secret crossing to a third-party service |
| PostHog response → rendered output | Untrusted external JSON must be validated before use |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-07 | Information Disclosure | POSTHOG_PERSONAL_API_KEY / POSTHOG_PROJECT_ID | mitigate | `import "server-only"`; both env vars server-only, never `NEXT_PUBLIC_`; documented as such in `.env.example` |
| T-09-08 | Injection | HogQL query strings | mitigate | Builders are static strings using `EVENT.*` constants; no interpolation of user/external input; `days` embedded as a numeric literal |
| T-09-09 | Tampering | PostHog response shape | mitigate | Zod `safeParse` on every response; return `null` (empty state) on mismatch — bad/injected data never reaches render |
| T-09-10 | Denial of Service | PostHog rate limit | mitigate | `next: { revalidate: 3600 }` caches results (~24 fetches/day), comfortably under the Query API limit |
| T-09-SC | Tampering | npm package installs | n/a | No package installs in this plan (recharts is installed in Plan 04, audit-approved in RESEARCH Package Legitimacy Audit) |
</threat_model>
