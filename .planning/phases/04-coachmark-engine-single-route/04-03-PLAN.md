<objective>
Build the load-bearing deliverable: `CoachmarkRenderer` wires react-joyride to the Phase-3 `useTour()` API and renders the engine that satisfies all four TOUR requirements on the single dashboard route. It reads `currentStepId`/`isActive`, drives joyride's controlled `stepIndex`, gates each step on its `data-tour` anchor being mounted (TOUR-01), renders a click-through cutout overlay above the z-[100] InstallGate via the `#tour-root` portal (TOUR-02), pauses while any Radix dialog is open (D-04, TOUR-02), positions within PWA safe-area insets (TOUR-03), and is fully keyboard/focus/aria-live/reduced-motion accessible (TOUR-04). It is loaded via `next/dynamic({ ssr: false })` (D-03) and mounted inside `TourProvider` in the `(tabs)` layout. A blocking human-verify checkpoint then confirms the engine against the UI-SPEC acceptance checklist, including on-device notched-standalone safe-area verification.

Purpose: This is the hardest technical surface (RESEARCH Pitfalls 1-3). Proving it on one route here means Phase 5 only adds content and cross-route sequencing, not engine work.
Output: `coachmark-renderer.tsx`, `coachmark-renderer-dynamic.tsx`, the `(tabs)/layout.tsx` mount, and a verified engine.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client overlay → app chrome | The coachmark overlay renders above ALL app chrome (nav, dialogs, InstallGate). A misconfigured overlay could trap input or hide critical UI. |
| third-party render → DOM | react-joyride mounts DOM via a portal and reads `data-tour` selectors; it executes in the client. |
| Radix dialog ↔ coachmark | Two overlay systems coexist; without pause logic the coachmark could sit over a dialog or steal focus. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Denial of Service | click-through overlay | mitigate | Overlay is `pointer-events: none` (TOUR-02) so it can never trap input or lock the user out; verified interactive in the human-verify checkpoint. App stays usable even if the tour is buggy. |
| T-04-06 | Denial of Service | anchor gating on unmounted target | mitigate | Renderer holds (renders null) until the `data-tour` target is in the DOM (TOUR-01) and degrades silently if it never mounts — no crash, no spotlight over empty space. |
| T-04-07 | Elevation of Privilege | Radix dialog focus coexistence | mitigate | Coachmark pauses while any `[role="dialog"][data-state="open"]` is present (D-04) and does not trap focus, so it cannot interfere with a modal's own focus management or block dismissal. |
| T-04-08 | Tampering | useTour() contract | accept | Renderer consumes the frozen 4-member TourValue unchanged; all persistence/authority stays in TourProvider + the owner-RLS PATCH route (Phase 1/3). The renderer writes nothing directly. |
</threat_model>
