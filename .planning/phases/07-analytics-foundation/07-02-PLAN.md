    - "use client" directive at top
    - Imports: `{ useEffect }` from `"react"`, `posthog` from `"posthog-js"`, `{ createClient }` from `"@/lib/supabase/browser"`
    - Export named function `PostHogIdentity()` returning `null`
    - Inside `useEffect` with empty deps `[]`:
      - Call `const supabase = createClient()`
      - Subscribe: `const { data: { subscription } } = supabase.auth.onAuthStateChange((event, session) => { ... })`
      - In the callback:
        - When `event === "SIGNED_IN"` and `session?.user` is truthy: call `posthog.identify(session.user.id)` — uses Supabase UUID as distinctId, never email or name (prevents PII leak)
        - When `event === "SIGNED_OUT"`: call `posthog.reset()` — clears the identified user from the PostHog client
      - Return cleanup: `return () => subscription.unsubscribe()`
    - No __loaded guard needed here — identify() and reset() are safe to call before init (PostHog queues them internally); however, if the team prefers the guard, it is acceptable to add it.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → PostHog (via /ingest proxy) | posthog.capture() sends event data; api_host points to /ingest reverse proxy |
| Supabase auth session → PostHog identity | session.user.id (UUID) is sent to PostHog as distinctId |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-03 | Information Disclosure | posthog-identity.tsx identify() call | mitigate | Use `session.user.id` (Supabase UUID) as distinctId — never `session.user.email` or display name; UUID is non-guessable and contains no PII |
| T-07-04 | Information Disclosure | NEXT_PUBLIC_POSTHOG_KEY in client bundle | accept | PostHog project API keys are intentionally public-facing (write-only ingestion key); secrets are server-side; this is by design per PostHog architecture |
| T-07-05 | Tampering | posthog-provider.tsx api_host env var | accept | NEXT_PUBLIC_POSTHOG_HOST is set at deploy time; value points to /ingest (local proxy) which rewrites to PostHog domains; no arbitrary host injection possible from client |
| T-07-06 | Spoofing | posthog.__loaded guard omitted | mitigate | Guard present in PostHogProvider useEffect prevents double-init; PostHogPageview also guards with !posthog.__loaded to prevent dropped events during StrictMode double-invoke |
</threat_model>
