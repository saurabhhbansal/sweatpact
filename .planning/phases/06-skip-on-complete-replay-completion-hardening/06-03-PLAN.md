<objective>
Build the "Pact is live" full-screen completion moment (UX-03) — the emotional
peak of v1.1. When the user has their first active challenge (D-01: the trigger
is challenge activation, NOT isTourComplete), a full-screen Radix Dialog takes
over /groups with sharp, consequence-first copy ("Your pact is live. / Real
money's on the line now. Show up — or pay up.") and a single "Let's go →" CTA.
It is shown exactly once — persisted via a `pact_live_seen` entry in
`completed_steps` (reuses the existing PATCH endpoint; the key satisfies
STEP_KEY_REGEX, is NOT in TEACHING_KEYS, and so never affects isTourComplete).
It works for both self-starter and invited paths because both land on /groups
with an active membership. The `/groups` RSC derives `hasActiveChallenge` from
its already-computed `activeMemberships` (no extra query) and passes it to the
client overlay.

Purpose: Satisfy UX-03 and Phase 6 success criterion 3 (a sharp, brand-voiced
"pact is live" moment marks first-challenge completion).
Output: New `pact-live-overlay.tsx` client component + a one-line wiring change
in the groups RSC.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| RSC → client overlay | hasActiveChallenge + completedSteps are computed server-side and passed as props; the client overlay only decides whether to render, it does not authorize anything |
| client overlay → PATCH /api/onboarding-progress | The pact_live_seen write crosses into the existing auth-gated, Zod-validated route handler |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-06 | Tampering | pact_live_seen seen-flag write | mitigate | Written via the existing PATCH endpoint whose PatchBody validates complete_step against STEP_KEY_REGEX; "pact_live_seen" is a benign cosmetic key that is NOT in TEACHING_KEYS, so a forged write cannot affect isTourComplete or any financial state |
| T-06-07 | Information Disclosure | completedSteps passed to client | accept | completed_steps is the viewer's own cosmetic onboarding progress (no PII, no financial data); it is already client-readable via the tour state. Low value |
| T-06-08 | Spoofing | hasActiveChallenge derivation | mitigate | Derived server-side from the viewer's own memberships (activeMemberships filtered to >=2 members within the RSC); the client cannot forge an active-challenge state to trigger the overlay, and the overlay has no financial side effect regardless |
| T-06-SC | Tampering | npm/pip/cargo installs | mitigate | No new package installs — Radix Dialog and lucide-react are already installed; no legitimacy gate needed |
</threat_model>
