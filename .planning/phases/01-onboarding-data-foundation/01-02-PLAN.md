logic and Zod validation extracted into a pure, unit-tested `src/lib/` helper. GET

## Artifacts This Plan Produces

- **Helper module:** `src/lib/onboarding-progress.ts`
  - `STEP_KEY_REGEX` — `/^[a-z0-9_]{1,40}$/`
  - `PatchBody` — Zod **strict** object: `complete_step?` (string matching `STEP_KEY_REGEX`),
    `last_step_id?` (string, same regex or nullable), `mandatory_done?` (boolean),
    `dismissed?` (boolean), `completed_at?` (ISO datetime string, nullable). `.strict()` rejects unknown fields.
  - `defaultProgress()` — returns the default-shaped GET object for the missing-row case
    (`{ mandatory_done:false, tour_version:1, last_step_id:null, completed_steps:[], dismissed:false, completed_at:null }`).
  - `mergeProgress(existing, patch)` — pure: dedupe-appends `patch.complete_step` into
    `existing.completed_steps`, applies optional scalar fields, returns the row to upsert.
    Replaying the same patch yields an identical result (idempotent additive append).
- **Test file:** `src/lib/onboarding-progress.test.ts` (Vitest)
- **Route file:** `src/app/api/onboarding-progress/route.ts`
  - `runtime = "nodejs"`, `dynamic = "force-dynamic"`
  - `GET()` — read caller's own row, defensive default
  - `PATCH(req)` — Zod-validated, idempotent upsert on `user_id`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → API route | Untrusted JSON body crosses into the server; must be Zod-validated before any write |
| API route → Postgres | Caller identity scoped via owner RLS; the route never uses service-role |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-06 | Tampering | unknown-field / malformed step-key injection via PATCH body | mitigate | `PatchBody` is `.strict()` (rejects unknown fields) with `STEP_KEY_REGEX` on `complete_step`/`last_step_id`; bad bodies → `validation_failed` 400 (unit-tested in Task 1) |
| T-01-07 | Spoofing / EoP | caller writes another user's progress | mitigate | Route uses non-admin `createClient()`; all DB ops `.eq("user_id", auth.user.id)` and upsert payload pins `user_id` to `auth.user.id`; owner-only RLS (plan 01-01) is the enforcement boundary |
| T-01-08 | Tampering | concurrent PATCH replay corrupts `completed_steps` | mitigate | Server-authoritative GET-then-dedupe-then-upsert via pure `mergeProgress`; replaying the same key is a no-op; client cannot send a full-array overwrite (D-04, Pitfall 4) |
| T-01-09 | Denial of Service | unauthenticated access | mitigate | `auth.getUser()` 401 guard on both GET and PATCH before any DB work |
| T-01-10 | Information Disclosure | GET leaks another user's row | mitigate | `.eq("user_id", auth.user.id)` + owner RLS; missing row returns a neutral default, never another user's data |
</threat_model>
