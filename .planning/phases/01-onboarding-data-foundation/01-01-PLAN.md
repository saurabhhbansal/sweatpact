<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → Postgres (via PostgREST) | Untrusted user identity asserts row ownership; RLS is the enforcement boundary |
| migration → existing trigger | `create or replace` of `handle_new_user()` can silently drop existing side effects |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | `onboarding_progress` cross-user read | mitigate | Owner-only RLS `select`/`update`/`insert`/`delete` policies (`auth.uid() = user_id`, `to authenticated`); verified in Task 2 via owner-scoped select returning only caller's row + `get_advisors` no-RLS-disabled check |
| T-01-02 | Elevation of Privilege | service-role bypass | mitigate | This phase uses NO admin/service-role client; only owner-scoped RLS governs the table (D-03, CONTEXT) |
| T-01-03 | Tampering | `create or replace handle_new_user()` drops profiles/profile_secrets inserts | mitigate | Task 1 acceptance criteria explicitly assert both prior inserts remain present after redefinition; verify grep confirms `insert into public.profile_secrets` still present |
| T-01-04 | Information Disclosure | backfill leaks rows across users | mitigate | Backfill is `insert ... select id ... from public.profiles` keyed per-profile with `on conflict (user_id) do nothing`; one row per existing user, no cross-user data copied |
| T-01-05 | Tampering | non-idempotent migration corrupts on re-run | accept→mitigate | Full idempotency (`if not exists`, `drop policy if exists`, `on conflict do nothing`, `create or replace`) makes re-application safe |
</threat_model>
