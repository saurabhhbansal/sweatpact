# Phase 3 ‚Äî UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

> **Scope note (read first). This phase ships NO new visual surface.** Phase 3 is invisible
> infrastructure (CONTEXT `<domain>` "Not this phase"):
>
> 1. Adds a username-only redirect **gate** to `src/app/(tabs)/layout.tsx` that redirects to the
>    **existing** `/onboarding/username` page when the username is missing/auto-generated (D-01, D-02).
> 2. **Removes** per-page `username` and `onboarding_complete` redirects from 8 existing tab pages
>    (D-03) ‚Äî no visual change, deleted logic only.
> 3. Creates `src/components/tour-provider.tsx` ‚Äî a `"use client"` React **context provider with NO
>    rendered output** (renders only `{children}`); mounted in the layout between nav shell and page
>    content (D-05).
> 4. Exposes the `useTour()` hook returning `{ currentStepId, isActive, advance, dismiss }` ‚Äî pure
>    state + behavior, **no UI** (D-07).
>
> **Nothing renders a coachmark, spotlight, overlay, or tooltip in this phase** ‚Äî that is Phase 4.
> The job of this contract is therefore twofold:
> (a) **Parity-lock** the existing `/onboarding/username` page so the new gate cannot regress it, and
> (b) **Freeze the `useTour()` context API as the interaction contract** that Phase 4+ coachmark
> surfaces will consume.
>
> Because there is no new pixel output, the six design dimensions are evaluated against
> **preservation** (no regression to the username page / app shell) rather than net-new design.

## Interaction Contract ‚Äî `useTour()` (the load-bearing deliverable)

This is the one **new** contract Phase 3 introduces. It is an API surface, not a visual one, but it is
the interaction contract every Phase 4+ coachmark consumer depends on. Freeze it exactly as below
(CONTEXT D-07).

**Hook:** `useTour()` returns:

| Member | Type | Behavior |
|--------|------|----------|
| `currentStepId` | `string \| null` | The next uncompleted, non-skippable step id; `null` when the tour is complete or dismissed. Derived from the Phase-2 pure probes (`lib/onboarding/completion.ts`) walking the ordered registry (`lib/onboarding/steps.ts`) ‚Äî D-09. |
| `isActive` | `boolean` | `true` exactly when `currentStepId !== null` (a step is pending and not dismissed). Coachmark surfaces gate their render on this. |
| `advance` | `(stepId: string) => Promise<void>` | Marks `stepId` complete, re-derives `currentStepId` from the pure probes client-side (no refetch), and persists via `PATCH /api/onboarding-progress` with `{ complete_step: stepId, last_step_id: stepId }` (D-08). Optimistic. |
| `dismiss` | `() => Promise<void>` | Sets `dismissed: true` via `PATCH /api/onboarding-progress` `{ dismissed: true }`; optimistically sets `currentStepId = null` and `isActive = false` (D-10). The single exit action ‚Äî there is no step-level skip in Phase 3. |

**Hydration contract (no-flash guarantee):**
- The `(tabs)/layout.tsx` RSC fetches progress server-side and passes it as `<TourProvider
  initialProgress={...}>` (D-04), mirroring the `GymOnboarding({ initialGymCount })` pattern from
  Phase 2. Consumers receive pre-hydrated state ‚Äî **no `useEffect` fetch, no client refetch flash.**
- `initialProgress` shape mirrors the Phase-1 `onboarding_progress` row:
  `{ mandatory_done, tour_version, last_step_id, completed_steps, dismissed, completed_at }`.
- **Fetch-failure = silent no-op (D-06):** if the server fetch fails (network/401/null),
  `initialProgress` is `null`; the provider treats this as a blank slate (`dismissed: false`,
  `completed_steps: []`) and the tour simply never activates. The app stays fully usable. **No error
  UI, no toast, no retry prompt is rendered.**

**Resume contract:** on reload mid-walkthrough, `currentStepId` re-derives to the same step from the
persisted `completed_steps` + probes ‚Äî no stored cursor needed (D-09).

**Deliberately deferred (do NOT add to the API in Phase 3 ‚Äî CONTEXT `<deferred>`):**
- `isComplete` and a `progress` shape ‚Äî deferred to Phase 4+ to avoid locking the API before the
  coachmark library is picked.
- Step-level skip ‚Äî only `dismiss()` exists in Phase 3 (D-10).
- The reset/replay path (`dismissed: false`, `last_step_id: null`) ‚Äî Phase 6.

## Gate / Redirect Interaction Contract

| Aspect | Contract |
|--------|----------|
| Gate location | `src/app/(tabs)/layout.tsx` only ‚Äî the single source of truth (D-01, D-03). |
| Trigger | Username missing or auto-generated (`isAutoUsername`: `!u \|\| /^user_[a-f0-9]{8}$/.test(u)`). |
| Action | `redirect("/onboarding/username")` ‚Äî server-side, before Suspense slots render so nav never flashes for ungated users. |
| Removed | `onboarding_complete ‚Üí /onboarding/schedule` redirect deleted everywhere (D-02). Users with optional setup incomplete now land in the real app. |
| Per-page redirects | Removed from `dashboard`, `groups`, `groups/[id]`, `cycle`, `notifications`, `settings`, `u/me`, `u/[username]` ‚Äî no per-page safety net (D-03). |
| Visual effect | None beyond the redirect itself. The `/onboarding/username` page renders **exactly as it ships today** (parity target above). The app shell (`TopNav` / `MobileNav` / header spacer) is unchanged. |
