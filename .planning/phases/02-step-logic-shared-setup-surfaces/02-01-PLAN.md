<objective>
Build the two pure, DB-free domain modules that own the walkthrough's "what to teach next" and "what's already done" decisions: an ordered step registry (`steps.ts`) with a `TOUR_VERSION` constant, and completion/skip probes (`completion.ts`) derived from real app state. Implements D-01 (presented = enough, four-key completion), D-02 (real-state probes, no duplicate flag), D-04 (uniform registry shape + TOUR_VERSION).

Output: `src/lib/onboarding/steps.ts`, `src/lib/onboarding/steps.test.ts`, `src/lib/onboarding/completion.ts`, `src/lib/onboarding/completion.test.ts`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller state → probe | Probes consume arbitrary caller-supplied arrays/numbers (gymCount, restDays, completedSteps). These are PURE functions; no privilege, no DB, no secrets cross this boundary. The only risk is logical: a wrong derivation could wrongly mark the tour complete or wrongly auto-skip a step. |
| registry → persistence (downstream) | Step ids produced here become `complete_step` keys Phase 1 persists. A malformed id (failing STEP_KEY_REGEX) would be rejected at the Phase-1 PATCH boundary — but we enforce the regex at the registry source so the contract never breaks. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | `steps.ts` STEPS array | mitigate | Freeze STEPS (`as const` / `Object.freeze`) and assert immutability in test (mirrors onboarding-progress.test.ts:26-31) so a downstream caller cannot mutate the shared registry. |
| T-02-02 | Tampering | step `id` → `complete_step` contract | mitigate | Test asserts every STEPS id satisfies the Phase-1 `STEP_KEY_REGEX`, so no id can produce a key the durable layer would reject or that smuggles oversized input. |
| T-02-03 | Information disclosure | probe purity | accept | Probes contain no secrets and touch no DB; they operate only on caller-supplied scalars/arrays. No PII or credential is reachable. Documented "no DB access" is enforced by the no-supabase-import grep gate. |
| T-02-04 | Elevation of privilege (logic) | `isTourComplete` four-key rule | mitigate | `isTourComplete` reuses `TEACHING_KEYS` (single source of truth) and is unit-tested for the 3-of-4 partial case, preventing a premature "complete" that would strand or over-advance a user. |
| T-02-SC | Tampering | npm/pip/cargo installs | n/a | No package-manager installs in this plan — both modules use only existing deps (zod is not even required for these pure modules; vitest is already installed). No `## Package Legitimacy Audit` needed. |
</threat_model>
