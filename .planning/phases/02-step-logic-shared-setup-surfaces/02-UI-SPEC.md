# Phase 2 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

> **Scope note (read first).** This phase is primarily PURE LOGIC (`lib/onboarding/steps.ts`,
> `lib/onboarding/completion.ts`) plus EXTRACTION of three EXISTING, already-styled setup
> components (`GymOnboarding`, `ScheduleForm`, `FinishOnboardingButtons`) into reusable,
> self-contained surfaces driven by an `onComplete` callback (CONTEXT D-03). This is **not
> net-new visual design**. The job of this contract is therefore **parity-locking**: the
> extracted surfaces must render and behave identically whether mounted by (a) the legacy
> linear `onboarding/*` route shell or (b) the walkthrough (Phase 3+). The contract below
> records the CURRENT shipping look/feel as the source of truth so extraction cannot regress
> it. **Coachmark / spotlight / overlay visuals are explicitly out of scope (Phase 4).**

## Spacing Scale

> **Parity-lock preamble (read first).** This is a parity-lock contract, not a new design. Every
> value listed under Exceptions below already exists VERBATIM in the shipping source and is being
> **PRESERVED, not introduced**. No new off-grid spacing is permitted in this phase. The 8-point
> grid governs any NEW spacing decision — but this phase makes none; it only reproduces what ships.

The codebase uses Tailwind's default 4px-based scale. The three surfaces use this subset today;
extraction must preserve it.

| Token | Value | Usage (as currently used in the three surfaces) |
|-------|-------|-------|
| 1 | 4px | `gap-1` not used; finest used gap is `gap-1.5` (6px) on day/goal pickers |
| 2 | 8px | `space-y-2` — label-to-control grouping; `gap-2` button internal icon gap |
| 3 | 12px | `gap-3` — skip-link ↔ primary-button row; `p-3` glass card padding; `px-3 py-2.5` list rows |
| 4 | 16px | `space-y-4` — gym surface section rhythm |
| 5 | 20px | `space-y-5` — schedule surface section rhythm |

Exceptions (PRESERVATION MANDATE — these values ship today and must be reproduced exactly; none are
new design choices, so the off-grid value below is not a grid violation introduced by this phase):
- `h-9 w-9` (36px) circular goal/rest-day toggle buttons (schedule surface) — fixed hit target, do not change.
- `h-11` (44px) Input height (`@/components/ui/input`) and `h-10` (40px) default Button height — primitive-owned, do not override.
- `gap-1.5` (6px) between picker chips — **pre-existing styling decision in `src/app/onboarding/schedule/client.tsx`** (the day/goal picker-chip clusters, lines 67 and 90: `flex flex-wrap gap-1.5`). This is an intentional dense cluster that ships today and is OFF the 8px grid. The extractor must reproduce it **exactly** — it is preserved, not introduced. NO new off-grid spacing may be added anywhere in this phase; this single value is grandfathered solely because it already ships.

> Parity rule: the `space-y-*` rhythm of each surface is part of its identity. When the surface is
> mounted inside the walkthrough vs. the legacy route, its OWN internal spacing is identical — only
> the wrapper/container around it differs (and that wrapper is owned by the mount context, not the surface).

## Typography

The app ships no custom type scale; sizes come from Tailwind utilities (`text-xs`, `text-sm`,
`text-base`). **This phase authors exactly two weights** (the type-scale decisions it owns):
400 for body and 600 for a direct Button label. The three surfaces use exactly the roles below —
extraction must not alter any size/weight.

**Surface-authored weights (the only weight decisions this phase makes):**

| Role | Size | Weight (authored) | Line Height | Where used |
|------|------|-------------------|-------------|------------|
| Body / control / helper | 14px (`text-sm`) / 12px (`text-xs`) | **400** (`font-normal`, default) | 1.5 body / ~1.4 caption | Input text, result `main_text`, primary status line, "Searching…", "N days per week.", validation + error lines, result `secondary_text` |
| Direct Button label | 14px (`text-sm`) | **600** (authored on/with the Button) | 1 (`inline-flex` centered) | "Continue", "Done", "Use my current location" |

> **This phase authors two weights only: 400 (body) and 600 (direct Button label). The 500 weight
> appears solely via the shadcn `<Label>` primitive's built-in contract and is not a scale decision
> here. Do not introduce any additional weight.**

**Primitive-owned weights (NOT authored by this phase — inherited from `@/components/ui/` contracts):**

These weights are baked into the existing shadcn/Radix primitives and arrive automatically when the
primitive is reused. They are primitive contracts, not type-scale choices of this phase, and must be
neither overridden nor counted as authored weights:

| Weight | Origin | Where it appears |
|--------|--------|------------------|
| 500 (`font-medium`) | `<Label>` primitive's built-in class; same class also on the picker-chip buttons | `<Label>` weekly goal / rest days; day/goal picker chips |
| 600 (`font-semibold`) | `@/components/ui/button` primitive's built-in default class | All `Button` default-variant labels (when the label is not separately authored at 600) |

Notes:
- Body line-height is globally `1.5` (set on `html, body` in `globals.css`). Headings n/a — these
  surfaces render no heading element of their own; the page/route shell owns any title above them.
- Net effect: the phase's own type scale is two weights (400 / 600). Any other weight you observe in
  the rendered output (e.g. 500) originates from a reused primitive's contract — preserve it, do not
  add to it, and do not treat it as a third authored display weight.

## Visuals

Each surface has ONE dominant visual anchor: the single solid-white primary CTA, which is the only
solid-white fill on an otherwise dark translucent-glass surface, drawing the eye to the one next action.
- **Gym surface** — anchor: the solid-white `Continue` button (against the dark glass search/results panel).
- **Schedule surface** — anchor: the solid-white `Continue` button (against the dark glass picker-chip clusters).
- **Shortcut surface** — anchor: the solid-white `Done` button (against the dark glass instruction panel).

Everything else (skip link, secondary actions, chips, helper text) stays in white-opacity ramps so the
primary CTA reads as the singular focal point in each surface. Preserve this hierarchy on extraction.

## Color

"Monochrome Glass" — a deliberately near-monochrome palette. Color is expressed almost entirely as
white-on-black with opacity ramps (`text-white/55`, `bg-white/[0.06]`, etc.). Accent hue is reserved
and essentially absent from these setup surfaces by design.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#000000` (`--background` `0 0% 0%`) | Page canvas, list dropdown background `bg-[#0a0a0a]` |
| Secondary (30%) | white at low alpha — `bg-white/[0.06]` controls, `.glass-card` (~4.5% white) | Inputs, picker chips, "gyms added" card, result list rows |
| Accent (10%) | `#FFFFFF` (`--primary` / `--accent` both `0 0% 100%`) | **Reserved list below** |
| Destructive | `0 72% 51%` (`--destructive`, a red) | Destructive Button variant — **not used in this phase** |

Accent (solid white) reserved for — explicit list, never "all interactive elements":
- The single **primary action button** per surface (`Button` default variant = `bg-white text-black`): "Continue" (gym, schedule) and "Done" (Shortcut).
- The **selected** goal chip and the active state of pickers (`bg-white text-black` selected goal; `bg-white/25 ring-white/40` selected rest day).
- Input **focus** border (`focus-visible:border-white`).

Everything else (skip link, secondary "Use my current location" button, helper text, result rows)
uses white-opacity ramps, never solid white fills — this is what keeps the one primary action visually singular.

Semantic state colors (already in theme, reused as-is, NOT recolored in this phase):
- `--success` (green) / `--warning` (amber) exist in the theme but these three surfaces deliberately
  render **error and validation text in `text-white/85`**, not red — preserve that choice (it is the
  current ship and part of the monochrome identity). Do not "improve" errors to red.

## Copywriting Contract

These are the EXACT strings currently shipping in the three components. Extraction must preserve them
verbatim so both mount contexts read identically. Any new string introduced by the `onComplete`/skip
wiring must match this voice (lowercase-calm, consequence-aware, no hype).

| Element | Copy |
|---------|------|
| Primary CTA — gym surface | `Continue` |
| Primary CTA — schedule surface | `Continue` (busy state: `Saving…`) |
| Primary CTA — Shortcut surface | `Done` (busy state: `Saving…`) |
| Secondary action — gym | `Use my current location` |
| Skip affordance (all three surfaces) | `Skip for now` |
| Empty state — gym (no gyms yet) | No card shown; the search Input placeholder `Search for your gym or address…` IS the empty state |
| Status — gym (after add) | `{count} gym{s} added.` / sub: `You can add more from Settings later.` |
| Empty/helper — schedule goal | `{goal} day{s} per week.` |
| Empty/helper — schedule rest days | `No rest days scheduled.` / when set: `{n} rest days · {7-n} gym days available` |
| Loading — gym search | `Searching…` |
| Error — gym search unavailable | `Search is unavailable right now. Use "Use my current location" for now.` |
| Error — gym maps not configured | `Search isn't configured. Use "Use my current location".` |
| Error — gym search generic | `Search failed. Try again.` |
| Error — gym geolocation absent | `Geolocation isn't available.` |
| Error — gym add failed | `{server error}` ?? `Failed to add gym.` |
| Validation — schedule over-budget | `Rest days + goal can't exceed 7.` (inline detail: `{n} rest days + {goal}-day goal = more than 7. Reduce one.`) |
| Error — schedule save failed | `{server error}` ?? `Failed` |
| Destructive confirmation | **None.** This phase has NO destructive action — no delete/reset/remove flows in any surface. |

**`onComplete` / skip interaction contract (the only new behavior — D-03):**

| Aspect | Contract |
|--------|----------|
| Skip placement | Left-aligned text-link (`text-sm text-white/55`, underline-on-hover), in a flex row at the bottom of the surface. |
| Primary placement | Right-aligned (`ml-auto`) solid-white `Button` in the SAME bottom flex row (`flex items-center gap-3`). |
| Skip behavior | Calls the same `onComplete`/skip path the mount context injects. Legacy route: `router.push(next)`. Walkthrough: advance-the-tour. The SURFACE never hard-codes a destination — it only invokes the injected callback. |
| Loading/disabled | While `busy`, BOTH skip and primary are `disabled` (`disabled:opacity-50`); primary swaps label to `Saving…` where that surface already does so (schedule, Shortcut). Gym keeps `Continue` but disables during `busy`. |
| Schedule extra-disable | Primary is additionally disabled when `tooMany` (rest days + goal > 7) — preserve. |
| Parity guarantee | The bottom action row (skip-left / primary-right, gap-3, same labels, same disabled rules) renders identically in both mount contexts. The mount context supplies only the callback, never the layout. |
