<objective>
Extract the three existing onboarding setup components (`GymOnboarding`, `ScheduleForm`, `FinishOnboardingButtons`) into self-contained, reusable surfaces under `src/components/onboarding/` driven solely by an injected `onComplete` callback (+ skip), then rewire the legacy `/onboarding/{gym,schedule,shortcut}` routes' `client.tsx` files into thin shells that pass `onComplete = router.push(next)`. The surfaces keep their own fetch+save against the EXISTING endpoints — they are NOT reduced to dummy presentational shells (D-03 "not dummy").

Purpose: This is the SETUP-01 "no logic fork" guarantee. The same surface that the legacy linear wizard mounts is the surface the walkthrough (Phase 3+) will mount with `onComplete = advance-the-tour`. Both paths hit identical save logic and identical existing endpoints. The Shortcut write-path is decoupled per Phase-1 D-05 so a walkthrough mount cannot prematurely flip `onboarding_complete`.

Output: 3 new surface components + 3 rewired legacy `client.tsx` shells. No new endpoints. No visual regression (parity-locked to 02-UI-SPEC.md).
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client surface → existing API routes | Each surface calls existing authenticated endpoints (`/api/places/search`, `/api/places/details`, `POST /api/gyms`, `PATCH /api/profile`, `PATCH /api/onboarding-progress`). All input validation, Zod, RLS, and rate limiting already live server-side in those routes — this phase adds NO new endpoint and weakens no existing guard. |
| surface vs shell write authority | The decouple boundary: the SURFACE may write `shortcut_viewed` (a benign progress key), but only the LEGACY shell may write `onboarding_complete: true`. This prevents a future walkthrough mount of the same surface from prematurely ending onboarding (Phase-1 D-05). |
| browser geolocation → /api/gyms | Geolocation coords are user-supplied and flow to `POST /api/gyms`; the existing route validates/stores them. Unchanged by this extraction. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05 | Elevation of privilege | ShortcutSurface write path | mitigate | The surface writes ONLY `complete_step: "shortcut_viewed"` via the existing Zod-`.strict()` PATCH (rejects unknown fields); it cannot write `onboarding_complete`. The `onboarding_complete` flip is confined to the legacy shell's `onComplete`, asserted by a grep gate (surface = 0 occurrences, shortcut shell ≥ 1). Prevents a walkthrough mount from ending onboarding early. |
| T-02-06 | Tampering | extracted save logic vs endpoints | mitigate | Extraction is verbatim; no endpoint is changed or added. Acceptance gates assert the surfaces reference only the pre-existing endpoint strings and the shells duplicate no save logic — the "no logic fork" guarantee (SETUP-01) is enforced mechanically. |
| T-02-07 | Information disclosure | client-side fetch error handling | accept | Surfaces surface server `error` strings into the UI (existing `text-white/85` pattern). These are the same non-sensitive error codes the routes already returned to these same components pre-extraction; no new disclosure. |
| T-02-08 | Spoofing / forged completion | financial-stakes adjacency | accept | This phase touches no check-in/penalty/settlement path. `onboarding_complete` and `shortcut_viewed` are onboarding-progress flags with no financial authority; RLS scopes both writes to the caller's own row/profile. No stakes are reachable from these surfaces. |
| T-02-SC | Tampering | npm/pip/cargo installs | n/a | No package-manager installs in this plan — all imports (`@/components/ui/*`, `next/navigation`, React) already exist. No `## Package Legitimacy Audit` required. |
</threat_model>
